# K线回放系统改进计划

## 1. 仓库分析

### 1.1 现有系统架构
- **前端**：React + TypeScript + Tailwind CSS
- **图表库**：TradingView Lightweight Charts
- **状态管理**：Zustand
- **后端**：Express.js
- **数据存储**：Parquet文件格式（本地缓存）
- **计算引擎**：mathjs（向量化计算）
- **遗传算法**：genetic-js

### 1.2 现有功能
- ✅ TradingView风格的K线图表
- ✅ 蜡烛图显示
- ✅ 鼠标点击选择起始点
- ✅ 本地Parquet缓存数据
- ✅ 基本回测功能
- ✅ 向量化回测引擎
- ✅ 遗传算法和因子挖掘

### 1.3 待改进功能
- ❌ K线回放效果需要更接近TradingView
- ❌ 回测功能需要支持在线API备用（AKShare/腾讯/新浪）
- ❌ 向量化回测需要优化以支持几秒钟回测5000只股票
- ❌ 遗传算法和因子挖掘功能需要进一步完善

## 3. 技术实现细节

### 3.1 K线回放改进
- 使用`lightweight-charts`的`timeScale()` API实现更精确的时间范围控制
- 实现平滑的动画效果，使用`requestAnimationFrame`进行帧控制
- 添加回放速度调节功能，支持0.5x到5x的速度范围

### 3.2 在线API集成
- **AKShare**：使用Node.js HTTP请求调用AKShare API
- **腾讯**：解析腾讯财经API的JSONP响应
- **新浪**：解析新浪财经API的CSV响应
- 实现API响应缓存，避免重复请求

### 3.3 向量化回测优化
- 使用mathjs的矩阵操作进行批量计算
- 实现数据分块处理，避免内存溢出
- 使用Promise.all进行并行处理
- 添加计算结果缓存机制

### 3.4 遗传算法改进
- 增加因子库，包括更多技术指标
- 优化适应度函数，考虑更多性能指标
- 实现因子相关性分析
- 添加因子重要性评估

## 5. 风险与应对策略

### 5.1 风险
- **API依赖风险**：在线API可能不稳定或有访问限制
- **性能风险**：回测5000只股票可能超出系统能力
- **内存风险**：处理大量数据可能导致内存不足

### 5.2 应对策略
- 实现API故障自动切换机制
- 优化数据处理和计算算法
- 实现数据分块处理和内存管理
- 添加系统监控和错误处理
